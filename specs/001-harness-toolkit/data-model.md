# Data Model: Harness Toolkit

## Memory Files (`.harness/memory/`)

These are the contract files between phases. Each file is markdown with structured sections.

### system-vision.md

| Field | Type | Required | Written By | Read By |
|-------|------|----------|------------|---------|
| System Name | text | yes | ideate | all |
| Problem Statement | paragraph | yes | ideate | domain, architect |
| Target Users | list | yes | ideate | domain, handoff |
| Desired Outcomes | list | yes | ideate | architect, handoff |
| Core Value Proposition | paragraph | yes | ideate | handoff |
| Scope — In Scope (v1) | list | yes | ideate | domain, architect |
| Scope — Out of Scope | list | yes | ideate | architect |
| Key Assumptions | list | yes | ideate | architect, research |
| Open Questions | list | no | ideate | clarify |

**Lifecycle**: Created by `ideate`, refined by `clarify`, read by all subsequent phases.

### domain-model.md

| Field | Type | Required | Written By | Read By |
|-------|------|----------|------------|---------|
| Bounded Contexts | list of sections | yes | domain | architect, handoff |
| Context — Name | text | yes | domain | architect |
| Context — Responsibility | paragraph | yes | domain | architect, handoff |
| Context — Core Entities | table (name, attributes, relationships) | yes | domain | architect, handoff |
| Context — Key Business Rules | list | yes | domain | handoff |
| Context Map | diagram/description | yes | domain | architect |
| Ubiquitous Language | glossary table (term, definition, context) | yes | domain | all |

**Lifecycle**: Created by `domain`, read by `architect`, `scaffold`, `handoff`.

### system-architecture.md

| Field | Type | Required | Written By | Read By |
|-------|------|----------|------------|---------|
| Architecture Style | enum (monolith/msa/hybrid) + rationale | yes | architect | scaffold, handoff |
| Services | list of sections | yes | architect | scaffold, handoff |
| Service — Name | text | yes | architect | scaffold, handoff |
| Service — Bounded Context | reference | yes | architect | handoff |
| Service — Responsibilities | list | yes | architect | handoff |
| Service — Communication | sync/async + protocol | yes | architect | research, handoff |
| Cross-Cutting Concerns | table (concern, approach, rationale) | yes | architect | research |
| Deployment Topology | description | no | architect | scaffold |

**Lifecycle**: Created by `architect`, read by `research`, `scaffold`, `handoff`.

### research-findings.md

| Field | Type | Required | Written By | Read By |
|-------|------|----------|------------|---------|
| Reference Systems | list (name, relevance, lessons) | yes | research | scaffold, handoff |
| Technology Recommendations | table (service, tech, rationale) | yes | research | scaffold, handoff |
| Infrastructure Concepts | list of sections (concept, recommendation, alternatives) | yes | research | scaffold |

**Lifecycle**: Created by `research`, read by `scaffold`, `handoff`.

## Phase Dependency Graph

```
ideate ──► system-vision.md
              │
clarify ──► system-vision.md (updated)
              │
domain ───► domain-model.md
              │
architect ─► system-architecture.md
              │
research ──► research-findings.md
              │
scaffold ──► (reads all, writes directories)
              │
handoff ───► services/*/spec.md
```

## Output Files (outside `.harness/memory/`)

### services/<name>/spec.md

Generated by `handoff`. Follows speckit's spec-template.md format:
- Feature name = service name
- User stories extracted from system vision + domain model
- Functional requirements from bounded context responsibilities + business rules
- Key entities from domain model
- Success criteria derived from system-level outcomes

These files are the bridge to speckit — each one is a valid speckit feature spec.
